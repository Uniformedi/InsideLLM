Decisions Locked — before any code lands

Two architectural decisions were locked. They shape every phase below.

Decision A · Locked

A3 — Hybrid visual surface

Form-based designer in Governance Hub at /governance/agents/new for compliance-authored agent manifests; Activepieces visual editor at /workflow/ for ops-authored workflow patterns.

Why: Manifests are flat policy documents, not DAGs. Different users, different surfaces. Agent designer matches the existing agent_manifest.schema.json exactly; Activepieces earns its keep without parallel UI work.
Decision B · Locked

B1 — Defer Managed Agents to 3.3

InsideLLM 3.2 ships without Anthropic Managed Agents support. 3.3 will add a runtime: managed_agents manifest field using a custom-tools-only pattern so every tool call round-trips through OPA/DLP.

Why: Managed Agents runs the agent loop on Anthropic's orchestration layer, bypassing LiteLLM/OPA/DLP. Shipping it in 3.2 would dilute the compliance narrative for the Parent Organization demo.

Phase 5 — Managed Agents Runtime 3.3 roadmap · post-Parent Organization

Architecturally compatible with InsideLLM's posture only if all agent tools are custom (client-side) so each tool call round-trips through the orchestrator → OPA → DLP before execution. Built-in bash / read / write / code_execution tools cannot be used without a perimeter exception.

Spec preview. Add runtime: managed_agents + custom_tools_only: true to the manifest schema. Governance Hub gets a session orchestrator (SSE consumer + custom-tool dispatcher). Each tool dispatch passes through OPA before executing locally; result returns to the Anthropic-hosted session as user.custom_tool_result. Vault credentials for any third-party APIs stay host-side. Prototype lands with the dispute-handler agent.

Risks & Contingencies

R1 · Phase 2B frontend overruns High
Likelihood: medium · Impact: demo slip

7d frontend budget assumes a single React engineer at full velocity. Any blocker (auth wiring, schema-to-form generator) cascades into demo prep.

If 2B.4 isn't shippable by 2026-05-04, freeze the designer at "form-only authoring + YAML I/O." Demo it as a draft, walk Parent Organization through the dispute-handler manifest in raw form. Land 2B.5 + 2B.6 in 3.2.1 two weeks later.
R2 · LiteLLM virtual key bootstrap idempotency High
Likelihood: medium · Impact: AP unable to call LLMs

null_resource running POST /admin/keys on every terraform apply creates duplicate keys, eventually exhausting LiteLLM's per-org key cap.

Use POST /admin/keys with a deterministic key_alias ("activepieces-vkey") and PUT semantics (delete-then-create on rotate). Stash the response in a Terraform-managed file via local_sensitive_file; triggers include the password hash so re-runs only fire on rotate.
R3 · pgvector extension not authorized in Postgres role Med
Likelihood: low (we control the DB) · Impact: pattern library DOA

CREATE EXTENSION vector requires superuser or membership in the trusted-extensions list.

The InsideLLM litellm Postgres role is a superuser per the existing compose. Verify on first apply; if not, run the CREATE EXTENSION as postgres in the init SQL (which always has full privs).
R4 · n8nempire corpus license unclear Med
Likelihood: low (commercial purchase) · Impact: redistribution claim

Patterns ship inside the InsideLLM image to customer deployments. Need explicit redistribution rights from n8nempire vendor.

Email vendor before merge. If unclear, ship pattern library empty; customer ingests their own corpus via POST /api/v1/patterns/ingest.
R5 · Activepieces healthcheck endpoint changed Low
Likelihood: low · Impact: container marked unhealthy, depends_on chain breaks

Draft uses /api/v1/flags. Pinning the image tag (Phase 1.1) locks the API surface — verify endpoint exists in the pinned version.

During Phase 1.1, smoke-test the pinned tag locally and confirm /api/v1/flags returns 200. If gone, switch to /api/v1/health or whatever the pinned version exposes.
R6 · OPA bypass on non-LLM AP steps Accepted (3.2)
Likelihood: high · Impact: governance gap

Workflow steps that hit external HTTP endpoints, write to DBs, or run code do not pass through OPA in 3.2 unless they call the insidellm-agent piece (4.1) or a deliberate governance.check step.

Phase 4.4 adds a minimal HTTP-out allowlist. Document the boundary in the 3.2 release notes. Full step-level OPA enforcement lands in 3.3 alongside Managed Agents.
