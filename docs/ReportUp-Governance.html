InsideLLM — ReportUp

Opt-in governance data sharing with a named parent organization.
Schema: v1.0 Auth: HMAC-SHA256 Chain: SHA-256 hash-linked envelopes Gate: compliance attestation required

ReportUp packs an organization's governance data into a cryptographically signed, tamper-evident envelope and ships it to a parent company's endpoint on demand or on a schedule. Every envelope is hash-linked to the previous one — the parent can detect gaps, tampering, or injected data at any point in the chain.

Envelope structure

Each run produces one envelope — a JSON document with these fields:

{
  "schema_version":           "1.0",
  "tenant_id":                 "<your-tenant-id>",
  "tenant_name":               "Acme Corp",
  "parent_name":               "Clarion Capital Partners",
  "generated_at":              "2026-04-22T02:00:00+00:00",
  "sequence_from":             1,
  "sequence_to":               42,
  "previous_envelope_hash":    "<SHA-256 of prior envelope>",  // chain link
  "audit_chain":               [ ... ],
  "telemetry":                 [ ... ],
  "agents":                    [ ... ],
  "identity_users":            [ ... ],
  "identity_groups":           [ ... ],
  "change_proposals":          [ ... ],
  "envelope_hash":             "<SHA-256 of canonical JSON above>",  // computed last
  "hmac_signature":            "<HMAC-SHA256(secret, envelope_hash)>"   // authentication
}

Chain-of-custody

previous_envelope_hash links each envelope to the one before it. A receiver that stores every envelope can detect:

  • Gaps — a missing envelope (sequence jump with no chain-hash match)
  • Tampering — any payload change breaks envelope_hash
  • Replay / injection — an inserted envelope breaks the chain link

Watermark

last_shipped_sequence records the highest audit sequence successfully acknowledged. Each run picks up from there. The watermark only advances after a parent ACK — a failed send is retried next run without skipping events.

Setup (first time)

Step 1 — Set the HMAC shared secret

Generate a secret and share it with the parent out-of-band:

openssl rand -hex 32   # produces 64 hex chars — use this value

In the ReportUp UI navigate to HMAC Secret and paste it, or via API:

curl -sk -X POST https://<host>/governance/api/v1/reportup/hmac-secret \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"secret": "<your-64-char-hex>"}'

Alternatively set REPORTUP_HMAC_SECRET in the environment before startup. The API value takes precedence.

Step 2 — Record an attestation

A compliance officer must attest before the feature can be enabled:

curl -sk -X POST https://<host>/governance/api/v1/reportup/attestation \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "attested_by": "jane.doe@example.com",
    "attestation_text": "I authorize sharing governance data with Clarion Capital Partners per our data sharing agreement dated 2026-04-01."
  }'

The response includes a snapshot_sha tied to the current config. This SHA must match when you enable ReportUp below.

Step 3 — Configure and enable

curl -sk -X PUT https://<host>/governance/api/v1/reportup/config \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "enabled": true,
    "parent_name": "Clarion Capital Partners",
    "parent_endpoint": "https://hub.clarion.example.com/insidellm/ingest",
    "share_audit_chain": true,
    "share_telemetry": true,
    "share_agents": true,
    "share_identity": false,
    "share_policies": false,
    "share_changes": true,
    "schedule_cron": "0 2 * * *",
    "max_records_per_run": 5000
  }'
409 Conflict? The config drifted since the attestation was recorded, or no attestation exists yet. Re-attest (Step 2) and try again.

Parent-side receiver

The parent endpoint receives a POST with these headers:

HeaderValue
Content-Typeapplication/json
X-Insidellm-Tenant<tenant_id>
X-Insidellm-Envelope-HashSHA-256 hex of the envelope
X-Insidellm-SignatureHMAC-SHA256 of the envelope hash
X-Insidellm-Schema-Version1.0

Verification (Python)

import hashlib, hmac, json

def canonical_json(obj: dict) -> bytes:
    return json.dumps(obj, sort_keys=True, separators=(",", ":"),
                      ensure_ascii=False).encode("utf-8")

def verify(envelope: dict, secret: str,
           expected_previous_hash: str | None = None) -> tuple[bool, str | None]:
    # 1. Recompute hash — exclude hash + signature fields from input
    pruned = {k: v for k, v in envelope.items()
              if k not in ("envelope_hash", "hmac_signature")}
    expected_hash = hashlib.sha256(canonical_json(pruned)).hexdigest()
    if envelope.get("envelope_hash") != expected_hash:
        return False, "envelope_hash mismatch — payload tampered"

    # 2. Verify HMAC
    sig = hmac.new(secret.encode(), expected_hash.encode(),
                   hashlib.sha256).hexdigest()
    if not hmac.compare_digest(sig, envelope.get("hmac_signature", "")):
        return False, "hmac_signature mismatch — wrong secret or tampered"

    # 3. Verify chain continuity
    if expected_previous_hash is not None:
        if envelope.get("previous_envelope_hash") != expected_previous_hash:
            return False, "previous_envelope_hash mismatch — chain broken"

    return True, None

The same logic is available in reportup_service.verify_envelope() for internal use.

ACK response

Return 200 OK with any JSON body. Any non-2xx response is treated as a rejection — the watermark does not advance and the run is logged as parent_rejected. ReportUp retries on the next scheduled run.

Scheduled runs

Set schedule_cron to a standard cron expression (UTC):

Security notes

  • The HMAC secret is stored in governance_settings_overrides and never returned via API — only the last 4 characters are shown for identification.
  • All ReportUp actions (config changes, attestations, secret rotations, runs) are written to the hash-chained audit trail.
  • Disabling ReportUp does not delete data. Re-enabling resumes from the last successful watermark.
  • max_records_per_run (default 5 000) caps each shipment. Backlogs drain across multiple successive runs, not one giant POST.

hmac_signature mismatch in log Shared secret out of sync Re-share secret via /hmac-secret on both sides
Sequence gap in parent log A run failed and was retried Normal — watermark re-sends the gap automatically
