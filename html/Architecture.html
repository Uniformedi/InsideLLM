Enterprise boundary

The platform is designed to collapse the AI consumer-facing surface and the AI governance plane into a single deployable boundary. Everything that happens between the user and the model traverses the host the customer controls.

Users, external API consumers (Claude Code CLI, custom apps), and inbound automation hit InsideLLM here. Nothing traverses a third-party cloud to reach the platform.

A single VM (or a fleet of them) runs Debian 12 Bookworm with Docker Compose. Hyper-V path is Terraform-provisioned; WSL2 path is PowerShell-scripted. Same image, same services on both.

Only the host's port 443 (and optionally 4000 for direct LiteLLM API access) crosses the enterprise network boundary. Everything else — all inter-service traffic, database access, policy evaluation — stays on the Docker bridge inside the VM.

Request flow

What happens when a user types a message in Open WebUI and presses Enter:

1
Browser → Nginx (port 443). TLS 1.2/1.3 termination. HSTS headers. auth_request to Governance Hub for JWT validation (OIDC / LDAP / Open modes).
2
Nginx → Open WebUI (:8080). OWUI applies any active pipelines (pre-DLP, OPA enforcement pipeline). Inlines uploaded file contents (PDF / DOCX / XLSX) into the message array so file content is scannable by DLP downstream.
3
OWUI → LiteLLM (:4000). OWUI calls LiteLLM's OpenAI-compatible /v1/chat/completions. Authentication uses a user-specific API key; metadata carries the agent identity (agent_id, tenant_id, guardrail_profile, manifest_hash).
4
LiteLLM: DLP inlet scan. configs/litellm/callbacks/dlp_guardrail.py scans user messages + inlined file content. SSN, credit card, PHI, credentials, custom regex patterns → block or redact depending on configuration.
5
LiteLLM: Humility guardrail (mandatory). humility_guardrail.py builds the OPA input (_build_opa_input) and evaluates. Humility alignment is checked; if OPA is unavailable, core Humility rules evaluate locally in Python (defense-in-depth fallback).
6
LiteLLM: industry OPA policy. If the agent's guardrail profile extends a tier (e.g., tier_fdcpa_regulated), the OPA bundle for that tier evaluates. Returns allow, deny_reasons, and obligations.
7
LiteLLM: budget + rate limit. PostgreSQL check for remaining daily budget; Redis atomic counter increment for RPM / TPM. Fail closed on exhaustion.
8
LiteLLM → Model provider. Request routed per allowed_models in the agent manifest. Cloud: api.anthropic.com / api.openai.com. Local: ollama:11434. BAA-covered routing restriction applies for PHI-tagged agents.
9
Model → LiteLLM. Response received (streaming or full).
10
LiteLLM: DLP outlet scan. Assistant response scanned before delivery. Any echoed-back PII / credentials / PHI blocked or redacted. Streaming: scan applies per-chunk on chunk boundaries.
11
LiteLLM: execute OPA obligations. audit.log, filter.fields, require.attestation, review.queue obligations fire as post-decision actions. Review-queue items land in Governance Hub's approval queue.
12
Audit record written. Governance Hub appends an entry to the hash-chained audit with sequence, previous_hash, chain_hash. Langfuse callback logs the full prompt / completion / cost / latency.
13
LiteLLM → OWUI → browser. Final response (post-DLP, post-obligation) streams back to the user.

Security architecture (layers)

Defense in depth. A failure at any single layer does not produce a governance escape.

Layer 1 — Prompt injection (soft)

configs/litellm/callbacks/humility_prompt.py — subclass of humility.adapters.litellm.HumilityPromptCallback. Injects alignment language into the system prompt. Cannot be disabled.

Layer 2 — Guardrail (hard)

configs/litellm/callbacks/humility_guardrail.py — evaluates core Humility rules in Python; delegates to OPA when available. Cannot be disabled.

Layer 3 — OPA policy engine

configs/opa/policies/humility/base.rego + industry overlays. Humility is the baseline; industry policies extend. Feature-flag disableable via policy_engine_enable — but Layers 1 + 2 still fire.

Layer 4 — Frontend pipeline

configs/open-webui/opa-policy-pipeline.py — additional frontend-side evaluation for OWUI-specific context. Swappable.

Cross-cutting — DLP gateway scanning

configs/litellm/callbacks/dlp_guardrail.py — inlet + outlet pattern-based scanning on all traffic, regardless of agent or tier. Independent of the Humility / OPA layers.

What exits the boundary vs. what stays

Stays inside the customer's network

User data, API keys, spend tracking, team membership, audit logs, RAG-indexed documents, Redis counters, PostgreSQL state, Humility policy sources, industry policy sources, Governance Hub configuration, hash-chained audit chain.

Leaves only if the customer routes to a cloud model

Per-request prompts and completions to api.anthropic.com or api.openai.com, scoped by the model provider's retention and privacy policy. Nothing leaves if the agent's allowed_models lists only Ollama models.

Observability data (customer-controlled)

Langfuse logs by default run on the customer's Langfuse instance (self-hosted). If the customer opts for Langfuse Cloud, logs traverse the boundary to Langfuse-controlled infrastructure. This is a customer configuration choice.

Extension points

Where customers add functionality without forking:

  • Custom OPA policies. Drop rego files into configs/opa/policies/tenants/<tenant>/; Governance Hub compiles and hot-reloads.
  • Custom DLP patterns. dlp_custom_patterns Terraform variable; regex-based; per-deploy.
  • Custom Open WebUI pipelines. Python files in configs/open-webui/pipelines/; registered via OWUI admin UI.
  • Custom tools / actions. Upload to /api/v1/actions/upload as YAML; scoped per-tenant in the action catalog.
  • Custom agent manifests. Author YAML; POST to /api/v1/agents/; publish via POST /{tenant}/{agent}/publish.
  • Custom LiteLLM callbacks. Python files in configs/litellm/callbacks/; registered in the LiteLLM config template.
  • External data connectors. PostgreSQL, MySQL, MSSQL, REST APIs configured through Governance Hub admin UI with RBAC, row filtering, field masking.
  • Custom DocForge templates. Drop templates into configs/docforge/templates/; referenced by agent actions.
