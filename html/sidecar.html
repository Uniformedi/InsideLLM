What it is

A browser extension delivered to endpoints via Windows Group Policy. It surfaces a side panel chat, right-click context actions (rewrite, summarize, translate), and an "ask about this page" action that includes the current tab's DOM as context. Every request terminates at your InsideLLM gateway; none goes to OpenAI or Anthropic directly from the browser.

Users get AI ubiquity across every tab they're in. Administrators get a single control plane: the same DLP, Humility, and audit that guard Open WebUI also guard every sidecar request, because the sidecar is just another client of the gateway.

Why a sidecar (vs. just pointing users at Open WebUI)

WorkflowOpen WebUISidecar
"Summarize this 40-page PDF I'm reading"copy-paste into chatright-click → summarize
"Rewrite this paragraph in Confluence"copy-paste → edit → paste backselection → accept inline
"What does this dashboard error mean?"screenshot → upload → askright-click → ask AI
Shadow-AI frictionquick tasks still tempt employees to public toolsin-browser ubiquity removes the "it's faster elsewhere" excuse

Open WebUI remains the deep-work surface — long conversations, RAG over documents, knowledge-base search. The sidecar is the everywhere-else surface.

Architecture

Employee's Chrome / Edge / Firefox
  │
  │  InsideLLM Assist extension
  │    - Side panel chat
  │    - Context menu actions
  │    - Page Q&A (reads current tab DOM as context)
  │    - Config locked: only talks to gateway URL below
  │
  │  HTTPS /v1/chat/completions
  │  Authorization: Bearer <user-virtual-key>
  ▼
InsideLLM gateway (on-prem VM / VLAN)
  │
  │  Nginx (TLS terminator, path-routed /v1/ → LiteLLM)
  │
  │  LiteLLM proxy callback chain:
  │    1. dynamic_rate_limiter_v3 (per-key tpm/rpm + budget)
  │    2. humility_prompt    (SAIVAS prompt injection)
  │    3. humility_guardrail (enforced rules; OPA-backed)
  │    4. dlp_guardrail      (inbound: PII/PHI/credentials/financials)
  │    5. forward to Anthropic / OpenAI / Bedrock / etc.
  │    6. dlp_guardrail      (outbound: redact echoed secrets)
  │
  │  Hash-chained audit → Governance Hub → fleet DB

The extension's HTTP base URL is pushed via Windows Group Policy (chrome.storage.managed). Users cannot change it from the extension settings when lockdown mode is on. Every request carries a per-user LiteLLM virtual key, also GPO-pushed. If the key is revoked in the Governance Hub, the next request fails with 401 — no tail of cached auth.

Configuration surface (chrome.storage.managed)

Defined in policy/managed-schema.json in the sidecar repo. Seven settings, all mapped 1:1 to the Windows ADMX/ADML templates in policy/gpo/:

Security posture

What the sidecar does NOT have

  • No vendor API keys. The extension never sees an Anthropic / OpenAI / Bedrock key. Only the gateway URL + a per-user virtual key.
  • No direct internet access to LLM providers. Traffic must terminate at your gateway.
  • No local storage of conversation content beyond the browser's standard extension storage. The authoritative audit record is on the gateway.
  • No way to disable DLP. The callback runs at LiteLLM regardless of what the client sends.

What the sidecar DOES inherit

  • DLP gateway scanning on every message in both directions (PII / PHI / credentials / financials + custom regex).
  • Humility alignment — SAIVAS-violating requests rejected before the model sees them.
  • OPA industry overlays (HIPAA, SOX, FERPA, GLBA, FDCPA, PCI-DSS) when enabled.
  • Per-user budgets and rate limits from the virtual key.
  • Hash-chained audit of every request in the Governance Hub.

Deployment guide (for administrators)

Prerequisites

  • Windows domain with GPO authority over target endpoints
  • A scratch Windows machine with Bun installed
  • An existing InsideLLM deployment reachable over HTTPS from endpoints
  • ssh + scp + gh CLI available

Step 1 — Clone + build

gh repo clone Uniformedi/page-assist-insidellm
cd page-assist-insidellm

.\scripts\fork.ps1          # apply overlay on upstream
cd build\upstream
bun install
bun run build               # produces build/chrome-mv3/

Step 2 — Sign

Sign the packed extension with your organization's signing key. Guard this key — changing it invalidates every installed instance and triggers Chrome's "extension corrupted" dialog. 

openssl genrsa -out signing-key.pem 2048  # once; keep in secret manager

chrome.exe --pack-extension="C:\...\build\chrome-mv3" `
           --pack-extension-key="signing-key.pem"
# → produces chrome-mv3.crx

Step 3 — Host the update channel on the InsideLLM VM

# on dev box
scp build/upstream/build/chrome-mv3.crx \
    insidellm-admin@<vm>:/tmp/insidellm-assist.crx

sed -e 's|{{EXTENSION_ID}}|<your-extension-id>|' \
    -e 's|{{GATEWAY}}|insidellm.corp.example.com|' \
    update/updates.xml.tpl > updates.xml
scp updates.xml insidellm-admin@<vm>:/tmp/

# on VM
sudo mv /tmp/insidellm-assist.crx /opt/InsideLLM/extensions/
sudo mv /tmp/updates.xml /opt/InsideLLM/extensions/

The nginx snippet at update/nginx-snippet.conf shows how to expose /extensions/ as a public read-only path. Add it to your nginx template.

Step 4 — GPO

  1. GPMC → Computer Configuration → Administrative Templates → Google → Google Chrome → Extensions.
  2. Enable Configure the list of force-installed apps and extensions. Entry format: 
    <extension-id>;https://<vm>/extensions/updates.xml
  3. Import the ADMX: 
    Copy policy/gpo/insidellm-assist.admx to %SystemRoot%\PolicyDefinitions\
Copy policy/gpo/insidellm-assist.adml to %SystemRoot%\PolicyDefinitions\en-US\
  4. Configure the seven settings under Administrative Templates → InsideLLM → InsideLLM Assist.
  5. Link the GPO to the target OU. Wait for Chrome's next policy refresh (~90 minutes) or run gpupdate /force on a test endpoint.

Step 5 — Verify on a test endpoint

  1. Chrome → chrome://policy — ExtensionInstallForcelist should show your entry.
  2. Chrome → chrome://extensions — extension present, marked "Installed by your administrator".
  3. Open the side panel — chat UI loads, no "add provider" controls.
  4. Send a test prompt. Confirm the request shows up in the Governance Hub audit log tagged with the virtual key's user identity.

End-user guide

Once installed, the extension adds:

  • Side panel button in the toolbar → opens chat panel
  • Context menu: right-click selected text for rewrite / summarize / ask / translate
  • Keyboard shortcut: Ctrl+Shift+L (configurable) opens the side panel

Chat

The side panel behaves like a mini Open WebUI: conversation history (stored locally in browser profile), model picker (restricted to the GPO allowlist), file attachments (if enabled).

Page Q&A

Toggle "Ask about this page" in the side panel to include the current tab's DOM as context. The DOM is sent to the gateway with the message, so DLP scans it. Large pages are summarized client-side before sending to stay within token budget.

What gets audited

Every message — inbound and outbound — is hash-chained and logged to the Governance Hub with user identity, timestamp, model, tokens consumed, DLP hits, Humility decisions. Users should assume everything they type is audit-logged.

Troubleshooting

SymptomCheck
Extension didn't install on endpointchrome://policy — force-install entry present? gpupdate /force run?
Extension installed but side panel emptychrome://extensions → Details → Errors. Common: gatewayBaseUrl missing in managed policy
"Network error" in chatFrom the endpoint, curl https://<gateway>/v1/models with the virtual key. 401 = key revoked; 502 = gateway down
"DLP blocked your message"Expected — user's message contained a pattern in the DLP ruleset. Rephrase and retry.
"This model is not available"Virtual key lacks access. Check the key's team in LiteLLM admin.
Key rotated but endpoints still using old oneChrome caches chrome.storage.managed. gpupdate /force + relaunch Chrome.

Roadmap

  • Build automation — CI that builds signed .crx on every release tag, signing key in GitHub Actions secret
  • Electron/Tauri desktop shell — for workflows outside the browser (IDE assistance, native apps). Lower priority; browser covers 80% of knowledge-worker AI use.
  • macOS / Linux deployment profile — managed-schema.json works on Chrome across platforms; just need per-OS distribution story (MDM on macOS, Chromebook enrollment, etc.)
  • Mobile — Chrome extensions don't run on mobile. A React Native app is the answer, not a port of this.
